# 产品需求文档 (Product Specification)

## 项目概述

**项目名称**：Quiet Space

**一句话描述**：一个有温度的AI日记Web应用，随时与AI朋友聊天，AI帮你把对话变成日记，并长期记住你的故事。

**目标**：打造一个不喧闹、不功利、有温度的数字陪伴空间。用户打开它不是为了"完成任务"，而是为了"被理解、被记住"。

**平台**：纯Web应用（浏览器访问），不依赖桌面客户端

**数据存储**：用户本地数据库（MySQL），通过本地后端服务连接

## 全局设计原则

### 视频背景是核心视觉层
- 整个产品基于HTML5 Video构建视觉语言
- 用户可上传视频作为背景，所有页面共享统一的视频背景
- 视频背景贯穿所有页面：首页、时间线、聊天、日记、设置

### 页面过渡必须有视觉过渡
- 所有页面切换（包括首页三个入口的进入）都需要视觉过渡动画
- 禁止生硬的硬切（hard cut）
- 过渡风格与产品整体氛围一致：缓慢、柔和、有呼吸感

### 氛围优先于功能
- 所有UI元素（按钮、弹窗、筛选器）的设计都要基于氛围
- 功能性UI不能破坏页面的沉浸感
- 深色、安静、有温度是所有视觉设计的底线

## 系统架构（四层模型）

```
┌─────────────────────────────┐
│     Persona Layer           │ ← AI人设（朋友/温柔姐姐/理性伙伴...）
├─────────────────────────────┤
│     Companion Layer         │ ← 陪伴策略（什么时候说/说多少/给不给建议）
├─────────────────────────────┤
│     Memory Engine           │ ← 记忆系统（提取/合并/召回/衰减/时间推理）
├─────────────────────────────┤
│     Data Layer              │ ← 数据库（diaries/chats/memories/emotionTimeline）
└─────────────────────────────┘
```

**核心设计原则：**

1. **记忆引擎是共享的**
   - 所有人格共享同一套记忆系统
   - 切换人格不会丢失记忆
   - 用户说"小王"，无论什么人格都知道小王是谁

2. **人设只影响表达方式**
   - Persona Layer 只负责"怎么说"
   - Memory Engine 负责"记得什么"
   - Companion Layer 负责"什么时候说、说多少"

3. **新增人格只需换 System Prompt**
   - 底层记忆系统不需要改动
   - 人格是可插拔的
   - 架构干净，扩展性强

## Memory Engine V2 架构（已实现）

```
User Message（用户输入）
        ↓
Memory Life Cycle Engine（记忆生命周期引擎）
        ├── Context Builder（上下文构建）✅
        ├── Thread State Engine（线程状态流转）✅
        ├── Emotion Update Engine（情绪趋势更新）✅
        └── Episodic Compression（情景记忆压缩）✅
        ↓
Working Memory（工作记忆）
        ↓
Companion Response（陪伴回复）
        ↓
Memory Extraction（记忆提取）
        ↓
Memory Consolidation（记忆合并）
```

**V2 核心变化：**

| 之前 | 之后 |
|------|------|
| 被动存储（Storage） | 主动演化（Living Memory System） |
| 手动调用模块 | 自动触发流水线 |
| 数据库 | 自我演化的状态系统 |

**V2 模块完成状态：**

| 模块 | 状态 | 核心能力 |
|------|------|----------|
| Context Builder | ✅ 已完成 | 从记忆中筛选最相关的内容 |
| Thread State Engine | ✅ 已完成 | 线程状态自动流转 |
| Emotion Update Engine | ✅ 已完成 | 情绪趋势更新 |
| Episodic Compression | ✅ 已完成 | 情景记忆压缩 |
| Memory Retrieval Ranking | ✅ 已完成 | 按相关性排序记忆 |
| Temporal Reasoning | ⏳ 待实现 | 区分不同时间的同类事件 |
| Thread Graph | ⏳ 待实现 | 事件链演化 |

**V2 问题清单：**

| 问题 | 状态 | 解决方案 |
|------|------|----------|
| VideoStorage 和 DB 重复 | 待修复 | 统一数据库连接 |
| 记忆检索性能问题 | 待优化 | 增加 tags 索引 |
| Memory Decay 不合理 | 待修复 | 新增 memoryCategory，identity 不衰减 |
| Recall Prompt 有风险 | 待优化 | 给一点来源感，不完全隐藏 |

| Memory Consolidation 太弱 | 待优化 | 优先依据 threadKey |
| 缺少 Context Builder | 待新增 | 最核心缺失，优先实现 |

## 首页入口

用户打开软件后，看到三个主入口，不直接进入任何功能，而是先选择"此刻想做什么"：

| 入口 | 图标建议 | 定位 | 映射功能 |
|------|---------|------|---------|
| 回忆 | 📖 | 看历史日记 | 功能4：日记编辑与浏览 |
| 倾诉 | 💬 | 跟AI对话，AI帮你整理成日记 | 功能1：聊天转日记 |
| 碎碎念 | 🧩 | 记录零碎想法，AI帮你组织成日记 | 功能8：碎碎念 |

**设计原则**：
- 三个入口视觉权重平等，没有主次之分
- 设置（API配置、AI人格、环境场景）通过齿轮图标进入，不占首页位置
- 首页氛围延续产品整体风格：深色、安静、有温度
- 进入各入口后可通过返回按钮回到首页

## 目标用户

**核心用户群体**：15-25岁学生/Z世代

**用户特征**：
- 年龄段：15-25岁
- 技术水平：有一定电脑操作能力，能按照引导配置API Key，但不需要编程知识
- 使用场景：任何想倾诉的时刻，感受、情绪波动、生活琐事。不想写正式日记但想记录下来。需要一个"不会评判自己"的倾听者。
- 痛点：
  - 想写日记但不知道怎么开始，面对空白页面无从下手
  - 情绪来了想倾诉但不想打扰朋友/不想被人知道
  - 写了日记但零零散散，事后回看不知道在说什么
  - 没有人长期记得自己说过的事、经历的变化

## 核心功能

### 功能 1：倾诉（Chat-to-Diary）

**功能描述**：用户和AI对话，AI在对话结束后自动生成日记草稿。用户不需要面对空白页面硬写，而是"先聊、再沉淀"。这是MVP的核心体验。进入倾诉后，用户通过3D粒子时间线展廊浏览历史对话，可继续旧对话或开启新对话。

**输入**：
- 用户对话内容：自然语言文本，多轮对话
- 对话时长：不限，通常5-15分钟
- 开题图片（可选）：用户上传一张图片作为对话背景和话题引子

**输出**：
- 日记草稿：富文本格式，包含标题、正文段落
- 情绪标签：AI从对话中提取的情绪关键词（如"疲惫""焦虑""期待"）
- 关键事件：对话中提到的人、事、时间
- 粒子背景：图片转化为3D粒子效果，作为对话的视觉背景

**业务规则**：

*时间线展廊*：
- 进入倾诉后，展示3D粒子时间线展廊，每个旧对话以粒子组成的图片形式展示
- 点击旧对话的粒子图片，弹出预览卡片，可选择"继续对话"或"查看日记"
- 时间线展廊与回忆、碎碎念共用视觉语言

*新对话开启*：
- 用户可选择上传一张图片作为"开题"，图片自动转化为粒子效果作为聊天背景
- 用户也可选择预设背景（壁炉、夜雨、星空等），预设背景后续补充
- 不上传图片且不选预设时，使用默认粒子效果
- 进入聊天界面后，由用户开启话题（AI不主动打招呼）

*对话与日记生成*：
- AI生成日记时，情绪强度不能偏离用户原话。用户说"有点难过"，AI不能写成"陷入绝望"。
- 日记标题由AI根据对话主题生成，用户可修改。
- 用户对草稿有完全编辑权：可以修改、重新生成、或丢弃。
- 对话记录完整保存，日记和对话双向关联（从日记能跳到原始对话）。
- AI在对话中不抢戏：先理解，再回应。不主动给解决方案，不长篇大论。
- 每段对话结束后，AI询问用户："要把今天的对话整理成日记吗？"用户选择"是"或"不用"。

**异常处理**：
- 用户对话过短（少于3轮）：AI提示"再多聊几句吧，这样我能帮你更好地整理。"
- AI生成日记质量差/偏离情绪：用户点"重新生成"，AI重新整理。
- 用户中途离开未保存：对话自动保存为草稿，下次打开可继续或转日记。
- 网络/API调用失败：提示用户检查网络和API配置，对话内容不丢失。
- 图片上传失败：使用默认粒子效果，不阻断对话流程。

**优先级**：高

**状态**：待开发

### 功能 2：基础记忆系统（Memory System）

**功能描述**：AI长期记住用户说过的人、事件、偏好和人生线程。让陪伴感从"每次重新认识"变成"它真的认识我"。

**核心理念**：
> 不是存储记忆，而是**什么值得记住，以及如何自然地记住**

> 用户最终会感动的不是 AI 回复得多聪明，而是某一天它说："那件困扰你两个月的事，终于结束了。"

> AI能记住事情，但只有少数AI能记住"事情发生在你人生的哪一段时间里"。

**记忆宪法 v1**：

| 原则 | 说明 | 示例 |
|------|------|------|
| **宁缺毋滥** | 只记对未来陪伴有价值的信息 | "今天吃了牛肉面"不记，"最近在准备考试"记 |
| **事实优先于推断** | 区分用户明确说的 vs AI 猜的 | "我讨厌加班"(confirmed) vs "最近天天熬夜"(inferred) |
| **人生线程优先** | 长期事件比单次事件更重要 | "考研""毕业设计""家庭关系" → thread 类型 |
| **情绪节点值得记** | 情绪变化是人生时间线的重要组成 | 长期焦虑后突然放松 → 重要记忆 |

**输入**：
- 用户对话/日记内容：自然语言文本
- 已有记忆库：用于合并/更新

**输出**：
- 事件记忆（event）：用户提到的具体事件（如"正在准备计算机考试"）
- 人物记忆（person）：用户提到的人及其关系（如"小王是大学室友"）
- 偏好记忆（preference）：用户明确表达的偏好（如"不喜欢被说教"）
- 人生线程（thread）：持续进行的事（如"计算机考试" - status: active/ongoing/resolved）

**核心 Prompt**：

*Prompt 1：Memory Extraction（记忆提取）*
- 触发时机：每轮对话结束后
- 输入：最近 20 条对话
- 输出：0-3 条新记忆（宁缺毋滥）
- 职责：识别值得记住的信息，生成记忆条目

*Prompt 2：Memory Consolidation（记忆合并）*
- 触发时机：提取新记忆后
- 输入：新记忆 + 已有记忆库
- 职责：合并相似记忆，更新人生线程状态
- Thread 状态：active → ongoing → paused → resolved

*Prompt 3：Memory Recall（记忆召回）*
- 触发时机：对话中相关话题出现时
- 输入：用户当前消息 + 相关记忆
- 输出：0-2 条召回建议（仅建议，不决定回复）
- 职责：识别哪些记忆相关，建议引用方式

*Prompt 4：Companion Response（陪伴回复）*
- 触发时机：生成每条回复时
- 输入：用户消息 + Persona + Recall建议 + 情绪上下文
- 输出：最终回复
- 职责：决定是否引用记忆、如何引用、回复风格

**业务规则**：

*记忆提取*：
- 每次最多新增 0-3 条记忆，避免记忆爆炸
- 重要性（importance）1-10，基于类型基础分 + 调整规则
- 置信度（confidence）0-1，区分 AI 推断 vs 用户确认
- 低置信度记忆需用户确认后才成为长期记忆

*记忆检索*：
- MVP 阶段：全量注入 Prompt，按 importance 排序
- 不需要向量数据库，数据量小，全量注入足够

*记忆引用*：
- Recall 只建议，Companion 人格决定是否引用
- 更新 lastReferencedAt 和 referenceCount
- 避免"记忆复读机"：referenceCount 越高，引用权重越低

*记忆衰减*：
- 记忆有生命周期：Create → Update → Reference → Decay → Archive
- `decayScore`：0-1，随时间自然衰减
- 衰减规则：每30天未引用，decayScore -0.1
- 被引用时：decayScore +0.2（但不超过1）
- decayScore < 0.3 的记忆自动归档（不主动引用）

*记忆管理*：
- 用户可查看 AI 记住了什么（记忆面板）
- 可手动删除不准确的记忆
- 可确认/取消 AI 的推断记忆

**异常处理**：
- AI提取了错误的记忆：用户可在记忆面板中删除或修正
- 记忆冲突（如用户前后说法矛盾）：AI以最新信息为准，但在适当时机温和确认
- 记忆数据量过大：设置保留策略，优先保留 isConfirmed=true 和 importance>=7 的记忆

**优先级**：高

**状态**：待开发

### 功能 3：情绪记录（Emotion Tracking）

**功能描述**：后台自动分析每段对话/日记的情绪，生成情绪数据。用户不需要手动打分，AI在背后默默记录。

**输入**：
- 用户对话内容或日记正文：自然语言文本

**输出**：
- 情绪标签：如"焦虑""疲惫""平静""开心"等
- 情绪强度：0-1的数值（如焦虑0.7，疲惫0.3）
- 心理活动：简短描述（如"挫败感""期待落空"）
- 情绪时间线：过去7天/30天的情绪趋势

**业务规则**：
- 分析在后台自动完成，用户感知不到分析过程
- 分析是为了"理解"，不是"诊断"。禁止输出"检测到抑郁倾向"等医学判断
- 每句话或每小段对话分析一次，情绪记录间隔不要太大
- 情绪数据可用于后续生成"本周情绪回顾"等总结
- 用户可以在情绪时间线上查看自己的情绪变化

**异常处理**：
- AI无法判断情绪：标记为"中性"或"未识别"，不强行分析
- 情绪分析明显偏差：用户可在情绪面板中手动修正标签

**优先级**：高

**状态**：待开发

### 功能 4：回忆（Journal Timeline & Editor）

**功能描述**：用户通过时间线浏览所有日记。时间线采用上下滑动交互，内容随位置淡入淡出，到视口中心时最清晰。右侧有竖直指示器显示当前位置。预览模式只显示摘要，点击查看详情。

**输入**：
- 日记内容：富文本格式（标题、段落、加粗、斜体、列表）
- 标签：用户手动添加或AI自动建议
- 筛选条件：情绪标签、来源、时间范围、颜色标签

**输出**：
- 时间线视图：日记按时间排列，上下滑动浏览
- 预览模式：只显示标题、日期、情绪标签、前100字摘要
- 详情页面：点击后显示完整内容
- 日记编辑器：全屏富文本编辑界面
- 筛选结果：符合筛选条件的日记在时间线上高亮/过滤显示

**业务规则**：

*时间线浏览*：
- 默认展示时间线视图，日记按时间垂直排列
- 采用上下滑动交互，内容随位置淡入淡出
- 到视口中心时最清晰（opacity: 1, scale: 1）
- 向边缘移动时逐渐淡出（opacity降低, scale缩小）
- 使用requestAnimationFrame优化滚动性能，确保流畅体验
- 首尾项目可滚动到屏幕中心（padding-top/bottom: 20vh）

*预览模式*：
- 每篇日记只显示：日期、标题、情绪标签、前100字摘要
- 显示"点击查看详情 →"提示
- 点击任意位置可查看详情页面

*右侧指示器*：
- 右侧有竖直排列的圆点指示器
- 当前位置金色高亮，其他位置半透明
- 点击可直接跳转到对应日记

*预览与编辑*：
- 点击时间线上的日记，进入详情页面
- 详情页面显示：标题、完整内容、情绪标签、来源标签、创建日期
- 详情页面提供操作：编辑、导出、删除
- 点击"编辑"进入全屏富文本编辑器
- 富文本编辑器支持：标题、段落、加粗、斜体、有序/无序列表

*筛选*：
- 筛选维度：情绪标签、来源（对话生成/手写）、时间范围、颜色标签
- 筛选不破坏时间线显示，作为氛围的一部分融入界面
- 筛选UI位置待定（需符合产品整体氛围设计）

*来源与关联*：
- 每篇日记自动关联：创建时间、情绪标签、来源（对话生成/手写，碎片整理归为手写）
- 日记可导出为Markdown文件
- 从日记可跳转到关联的原始对话（对话生成的日记）或碎片（碎片整理的日记）

**异常处理**：
- 编辑未保存关闭：自动保存为草稿
- 导出失败：提示用户检查磁盘空间和权限
- 时间线数据量过大：按月分组加载，避免一次性渲染过多日记

**优先级**：高

**状态**：已完成（时间线滑动效果已实现，视频背景已应用）

### 功能 5：AI人格系统（AI Personality）

**功能描述**：提供预设AI人格选择，同时支持用户自定义调节AI的行为参数。AI还能通过长期对话自适应用户的偏好。

**输入**：
- 人格选择：预设人格（朋友/温柔姐姐/理性伙伴/安静倾听者/轻松幽默）
- 参数调节：主动性、回复长度、安慰程度、追问程度、建议倾向（滑块0-100）
- 自适应开关：用户选择是否允许AI学习自己的偏好

**输出**：
- AI回复风格的变化：根据人格和参数调整语气、长度、主动性
- 人格配置持久化：用户的设置保存到本地

**业务规则**：
- 预设人格是参数的快捷方式，选了预设后用户仍可微调具体参数
- AI自适应默认关闭，用户主动开启后AI才会学习用户偏好（如"用户不喜欢大道理""更喜欢简短回应"）
- 自适应调整幅度温和，不会突然改变风格
- 用户可随时重置为默认人格
- 人格设置对所有对话和日记生成生效

**异常处理**：
- 自适应导致AI行为偏离用户预期：用户可关闭自适应并重置
- 参数调节后效果不明显：提示用户某些参数需要几轮对话后才能体现变化

**优先级**：高

**状态**：待开发

### 功能 6：环境场景（Ambient Scene）

**功能描述**：为所有页面提供沉浸式视频背景。用户可上传自定义视频作为背景，所有页面共享统一的视频背景效果。

**输入**：
- 视频文件：用户上传的MP4/WebM视频
- 透明度调节：视频背景透明度（0-100%，默认60%）

**输出**：
- 视频背景：全屏视频循环播放，覆盖所有页面
- 视觉效果：视频以object-fit: cover模式显示，保持比例，裁剪边缘

**业务规则**：
- 视频背景在最底层，UI内容浮在其上，两者分离
- 所有页面共享同一视频背景，保持视觉一致性
- 视频自动循环播放，无控制按钮
- 视频静音播放（muted），避免干扰用户
- 视频透明度默认60%，确保内容可读性
- 支持MP4、WebM、OGG格式
- 视频文件存储在backgrounds目录

**异常处理**：
- 视频文件缺失：显示纯黑背景，不提示错误
- 视频格式不支持：显示纯黑背景，控制台输出错误
- 视频加载失败：显示纯黑背景，不影响功能使用

**优先级**：中

**状态**：已完成（视频背景已应用到所有页面）

**异常处理**：

**异常处理**：

- 数据迁移失败：自动回滚，提示用户检查旧数据
- 数据库版本不兼容：提示用户升级数据库或提供兼容方案

### 功能 8：碎碎念（Fragment Notes）

**功能描述**：用户记录零散想法的地方。不同于"倾诉"的实时对话，碎碎念是用户自己打字记录脑子里的碎片念头，AI在旁边默默整理，帮用户把碎片拼成完整日记。支持时间线可视化浏览，碎片可单独或批量总结。

**输入**：
- 碎片内容：自然语言文本，用户随手记录
- 情绪标签：用户手动选择（emoji/文颜字/艺术图标），AI可建议但需用户确认
- 颜色标签：用户可选颜色标记
- 总结方式：用户选择"单独总结"或"批量合成总结"

**输出**：
- 碎片列表：按时间排列的碎片记录
- AI整理预览：结构化的日记草稿（用户确认后转为日记）
- 情绪标签：带颜色和艺术化emoji的情绪标记
- 时间线视图：碎片在时间轴上的可视化分布

**业务规则**：

*创建与浏览*：
- 打开碎碎念页面，默认显示已有碎片列表
- 用户可随时创建新碎片
- 每个碎片是独立实体，支持分多次续写
- 碎片可单独查看、编辑、删除

*AI整理触发*：
- 用户主动：点击"帮我整理"按钮，AI整理当前碎片
- 自动静默：用户停笔超过90秒（可配置：30秒-5分钟），AI在旁边弹出预览小面板，展示整理结果
- 两种方式产出的整理结果，都由用户决定是否转为日记

*批量总结*：
- 用户可选择多个碎片或时间片段
- 选择"分别总结"：每个碎片各自生成一篇日记
- 选择"合成总结"：多个碎片合并为一篇日记，AI按时间排序理解上下文
- 所有AI产出需用户确认后才转为日记

*情绪标签*：
- 支持emoji、文颜字（如(╯°□°)╯︵ ┻━┻）、自定义下载的艺术图标
- AI分析碎片内容后建议情绪标签，用户可接受、修改或拒绝
- 情绪标签可用于后续筛选和情绪时间线

*数据持久化*：
- 未整理的碎片保留在碎碎念列表中，用户可随时回顾
- 已转日记的碎片标记"已整理"，仍保留在列表中
- 碎片与日记双向关联（从碎片可跳到对应日记）

**异常处理**：
- 用户长时间未完成碎片：自动保存为草稿，不丢失
- AI整理结果不符合预期：用户可重新生成或手动编辑
- 批量总结碎片过多：限制单次最多20个碎片，超出提示分批处理
- 情绪标签下载失败：回退到内置emoji列表

**优先级**：中

**状态**：待开发

## 功能优先级

### 高优先级（MVP必须有）
1. **数据库连接** - 数据存储基础，没有这个其他功能无法工作
2. **API配置** - 没有这个用户无法使用AI功能
3. **聊天转日记** - 产品核心体验，验证"用户愿不愿意用AI帮写日记"
4. **基础记忆系统** - 陪伴感的基础，"它真的记得我"
5. **情绪记录** - 后台自动分析，为后续情绪时间线打基础
6. **日记编辑与浏览** - 日记的查看、编辑、搜索
7. **AI人格系统** - 至少有预设人格选择，自定义可简化

### 中优先级（第一版尽量有）
1. **环境场景（壁炉夜谈/夜雨窗边）** - 沉浸式氛围背景
2. **碎碎念** - 碎片记录+AI整理，MVP先做2D版本，后续升级3D时间线
3. **日记导出** - 备份需求，用户安全感
4. **情绪时间线可视化** - 情绪记录的展示层
5. **IndexedDB存储升级** - 替代localStorage，容量大、支持二进制数据、事务安全

### 低优先级（后续版本）
1. **照片粒子背景** - 用户上传照片作为粒子氛围背景
2. **AI看图对话** - AI识别照片内容开启对话，需多模态API
3. **陪伴写日记模式（双栏模式）** - 第二阶段
4. **语音功能（AI语音回复+语音输入）** - 第二阶段
5. **AI来信** - 偶尔主动关心用户，产品惊喜点
6. **写给未来自己（时间胶囊）** - 第三阶段
7. **更多环境场景** - 根据用户反馈扩展
8. **云端同步** - 用户有明确需求时再做
9. **移动端适配** - 响应式布局，支持手机浏览器

## AI增强功能

整个产品本身就是AI驱动的，以下是核心AI能力：

### AI增强 1：对话→日记生成

**AI能力描述**：将用户的多轮对话自动整理成一篇结构完整、情绪准确的日记草稿

**触发条件**：用户结束一轮对话后，主动选择"整理成日记"

**输入**：完整对话记录 + 用户的人格偏好设置

**输出**：富文本日记草稿（标题+正文+情绪标签）

**示例**：
- 用户对话："今天项目延期了，很烦""感觉努力白费了""晚饭倒是挺好吃的"
- AI输出标题：《有些努力没有回音的一天》
- AI输出正文：整理成连贯段落，保留"烦""挫败"的情绪，不夸大也不弱化

---

### AI增强 2：情绪自动分析

**AI能力描述**：从对话/日记文本中自动提取情绪标签和强度，无需用户手动标注

**触发条件**：每段对话结束或每篇日记保存时

**输入**：对话/日记文本

**输出**：情绪标签+强度值+心理活动描述

**示例**：
- 输入："今天特别烦，项目没推进，但晚饭挺好吃"
- 输出：焦虑0.6，烦躁0.7，疲惫0.3 | 心理活动：挫败感，小确幸

---

### AI增强 3：长期记忆提取

**AI能力描述**：从对话/日记中自动提取关键事件、人物、情绪模式，构建用户记忆库

**触发条件**：每次对话/日记处理完成后

**输入**：对话/日记文本 + 已有记忆库

**输出**：新增/更新的记忆条目

**示例**：
- 用户说："小明今天又迟到了"
- AI记忆：小明 = 室友/同学，有迟到习惯，2026-05-26记录

---

### AI增强 4：日记润色

**AI能力描述**：用户写了一篇零散的日记后，AI帮忙整理结构、优化表达，但不改变情绪和事实

**触发条件**：用户主动请求"帮我润色"

**输入**：用户原始日记文本

**输出**：润色后的日记（结构更清晰，表达更流畅）

**示例**：
- 原文："今天烦。项目没推进。晚饭好吃。有点累。"
- 润色："今天的情绪有些复杂。项目推进不顺利，让我感到烦躁和疲惫。不过晚饭还不错，算是一天里难得的小安慰。"

#### 数据库表结构（MySQL）

```sql
-- 日记表
CREATE TABLE diaries (
  id VARCHAR(36) PRIMARY KEY,
  title VARCHAR(255) NOT NULL,
  content TEXT,
  emotions JSON,
  date DATETIME,
  source ENUM('chat', 'manual', 'fragment') DEFAULT 'manual',
  messages JSON,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

-- 对话表
CREATE TABLE chats (
  id VARCHAR(36) PRIMARY KEY,
  title VARCHAR(255),
  messages JSON,
  date DATETIME,
  emotion_summary JSON,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

-- 碎碎念表
CREATE TABLE fragments (
  id VARCHAR(36) PRIMARY KEY,
  text TEXT,
  date DATETIME,
  emotion VARCHAR(50),
  organized BOOLEAN DEFAULT FALSE,
  diary_id VARCHAR(36),
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (diary_id) REFERENCES diaries(id)
);

-- AI记忆表
CREATE TABLE memories (
  id VARCHAR(36) PRIMARY KEY,
  type ENUM('event', 'person', 'preference', 'thread') NOT NULL,
  content TEXT NOT NULL,
  importance INT DEFAULT 5,
  confidence DECIMAL(3,2) DEFAULT 0.5,
  is_confirmed BOOLEAN DEFAULT FALSE,
  source_id VARCHAR(36),
  tags JSON,
  last_referenced_at DATETIME,
  reference_count INT DEFAULT 0,
  decay_score DECIMAL(3,2) DEFAULT 1.0,
  status ENUM('active', 'archived') DEFAULT 'active',
  temporal JSON,
  thread_key VARCHAR(100),
  instance VARCHAR(100),
  parent_thread_id VARCHAR(36),
  memory_category ENUM('identity', 'relationship', 'project', 'event', 'emotion') DEFAULT 'event',
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

-- 情绪时间线表
CREATE TABLE emotion_timeline (
  id VARCHAR(36) PRIMARY KEY,
  timestamp DATETIME,
  source_id VARCHAR(36),
  source_type ENUM('chat', 'diary', 'fragment'),
  emotions JSON,
  dominant_emotion VARCHAR(50),
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 设置表
CREATE TABLE settings (
  setting_key VARCHAR(100) PRIMARY KEY,
  setting_value TEXT,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

-- 索引
CREATE INDEX idx_diaries_date ON diaries(date);
CREATE INDEX idx_chats_date ON chats(date);
CREATE INDEX idx_fragments_date ON fragments(date);
CREATE INDEX idx_memories_thread_key ON memories(thread_key);
CREATE INDEX idx_memories_status ON memories(status);
CREATE INDEX idx_emotion_timeline_timestamp ON emotion_timeline(timestamp);
```

#### 记忆类型说明

| type | 说明 | 示例 |
|------|------|------|
| `event` | 具体事件 | "用户正在准备计算机考试" |
| `person` | 人物关系 | "小王是用户大学室友" |
| `preference` | 用户偏好 | "用户不喜欢被说教" |
| `thread` | 人生线程（持续进行的事） | "计算机考试" - status: active/ongoing/resolved |

#### 记忆字段说明

| 字段 | 说明 |
|------|------|
| `importance` | 1-10，基于类型基础分 + 调整规则 |
| `confidence` | 0-1，AI 对这条记忆的置信度 |
| `isConfirmed` | 用户是否明确确认过这条记忆（区分 AI 推断 vs 用户确认） |
| `lastReferencedAt` | 最后被 AI 在对话中引用的时间 |
| `referenceCount` | 被 AI 引用的次数，避免"记忆复读机" |
| `decayScore` | 0-1，记忆衰减分数，随时间自然衰减（identity 类不衰减） |
| `status` | 记忆状态：active/archived |
| `temporal` | 时间上下文：{startDate, endDate, isRecurring, periodLabel, timeBucket} |
| `threadKey` | 线程标识符（如"computer_exam"），用于识别同类事件 |
| `instance` | 线程实例（如"2026_spring"），区分不同时间的同类事件 |
| `parentThreadId` | 父线程 ID，用于构建事件链 |
| `memoryCategory` | 记忆分类：identity/relationship/project/event/emotion |

#### 三层记忆结构

| 层 | 说明 | 示例 | 衰减 |
|---|------|------|------|
| **Semantic Memory** | 长期事实 | "用户叫唐彬程"、"用户是计算机专业" | identity 不衰减 |
| **Episodic Memory** | 具体经历 | "2026-06-16 第一次完成记忆系统" | 正常衰减 |
| **Emotional Memory** | 情绪节点 | "长期焦虑后突然放松" | 正常衰减 |

| 分类 | 说明 | 衰减规则 |

| `event` | 事件相关 | 正常衰减 |
| `emotion` | 情绪相关 | 正常衰减 |

### 术语表
- **OpenAI兼容接口**：遵循OpenAI Chat Completions API格式的接口，大多数大模型都支持
- **富文本**：支持格式（加粗、斜体、标题等）的文本编辑，区别于纯文本
- **情绪光谱**：用多个情绪维度（而非单一"开心/难过"）来描述用户情绪状态
- **RAG（Retrieval-Augmented Generation）**：检索增强生成，先从记忆库检索相关信息，再让AI基于这些信息生成回复
- **HTML5 Video**：HTML5原生视频元素，支持autoplay、muted、loop等属性
- **object-fit: cover**：CSS属性，保持视频比例，裁剪以覆盖容器
- **Ambient（环境层）**：产品的氛围底层，包括视频背景和视觉效果
- **IndexedDB**：浏览器内置的本地数据库，支持存储大容量数据（视频、图片等二进制文件）
- **localStorage**：浏览器本地存储，适合存储小量文本数据（设置、日记等），容量约5-10MB
- **Blob**：二进制大对象，用于存储文件数据（如视频、图片）
- **requestAnimationFrame**：浏览器API，用于在下一帧重绘前执行动画，确保60fps流畅动画
- **CSS Variables**：CSS变量，用于统一管理颜色、间距等设计token，便于主题切换
- **MIME类型**：Multipurpose Internet Mail Extensions，用于标识文件类型（如video/mp4）

### 交互动态术语
- **CSS Transition**：CSS过渡，定义属性变化的持续时间和缓动函数，实现平滑的状态切换
- **CSS Animation**：CSS动画，通过@keyframes定义关键帧，实现复杂的动画效果
- **Transform**：CSS变换，包括平移（translate）、旋转（rotate）、缩放（scale）、倾斜（skew）
- **Opacity**：透明度，值从0（完全透明）到1（完全不透明），用于淡入淡出效果
- **Fade In/Out**：淡入/淡出，通过opacity变化实现元素的渐显或渐隐
- **Scroll**：滚动交互，用户通过鼠标滚轮或触摸滑动浏览内容
- **Drag and Drop**：拖拽交互，用户拖动元素到目标区域完成操作（如上传文件）
- **Hover**：悬停效果，鼠标移到元素上时触发的视觉反馈
- **Easing Function**：缓动函数，控制动画速度变化曲线（如ease、cubic-bezier）
- **Parallax**：视差效果，不同层次元素以不同速度移动，创造深度感
- **Will-change**：CSS属性，提示浏览器该元素将发生动画变化，提前优化渲染
- **Throttle**：节流，限制函数执行频率，避免滚动等高频事件导致性能问题

### 高级动效术语（参考KTV项目）
- **Clip-path**：裁剪路径，用inset/circle/polygon等形状裁剪元素，实现各种填充动画效果
- **Jelly Bounce**：果冻弹跳，通过非等比scale变换（如scale(1.25, 0.8)）模拟弹性效果
- **Particle Effect**：粒子效果，动态创建小元素并分散动画，用于点击反馈
- **Radial Gradient**：径向渐变，从中心向外扩散的颜色渐变，用于光晕效果
- **Keyframes**：关键帧，定义动画各阶段的状态，如0%、50%、100%
- **cubic-bezier**：三次贝塞尔曲线，自定义缓动函数，如(0.34, 1.56, 0.64, 1)实现弹性效果
- **Fill Animation**：填充动画，通过clip-path变化实现从无到有的填充效果
- **Pop Glow**：弹出光晕，动画完成后出现的光环扩散效果
- **Shrink Pop**：收缩弹跳，元素先缩小再恢复，用于取消操作反馈
- **Delay**：延迟，动画开始前的等待时间，用于错开多个动画的执行时机
- **Forwards**：填充模式，动画结束后保持最后一帧的状态
- **DOM Manipulation**：DOM操作，动态创建/删除页面元素（如粒子）
